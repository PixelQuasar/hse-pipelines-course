## 1. Конкурентная запись и ACID

| Формат | Результат | Механизм |
| :--- | :--- | :--- |
| **Delta Lake** | Успешно | Optimistic Concurrency Control |
| **Iceberg** | Успешно | Optimistic Concurrency Control |
| **Hudi** | Успешно | MVCC + Optimistic Locking |

**Вывод**: Все форматы поддерживают ACID и корректно обрабатывают конкурентный append

## 2. Производительность

### 2.1. Размер на диске

| Формат | Размер (MB) | Комментарий |
| :--- | :--- | :--- |
| **Delta Lake** | ~3.5 MB | Использует Parquet + _delta_log |
| **Iceberg** | ~3.5 MB | Использует Parquet + metadata |
| **Hudi** | ~4.5 MB | Использует Parquet + .hoodie. Hudi может занимать больше места из-за дополнительных метаданных для индексации и управления версиями. |


### 2.2. Скорость чтения

| Формат | Mean Time (ms) | OPS (Operations Per Second) |
| :--- | :--- | :--- |
| **Iceberg** | ~32 ms | ~31.3 |
| **Hudi** | ~77 ms | ~13.0 |
| **Delta Lake** | ~133 ms | ~7.5 |

**Анализ**:
*   **Iceberg** показал лучшую производительность на чтение в данном тесте. Это может быть связано с эффективным пропуском данных с помощью использования метаданных
*   **Hudi** оказался посередине
*   **Delta Lake** показал худший результат в данном конкретном тесте на локальной машине

### 2.3. Скорость записи

| Формат | Mean Time (ms) | OPS |
| :--- | :--- | :--- |
| **Iceberg** | ~725 ms | ~1.38 |
| **Delta Lake** | ~1160 ms | ~0.86 |
| **Hudi** | ~3530 ms | ~0.28 |

**Анализ**:
*   **Iceberg** оказался самым быстрым на запись
*   **Delta Lake** немного медленнее Iceberg
*   **Hudi** значительно медленнее на запись

## 3. Кейсы использования

### Когда оправдано использовать Hudi вместо Iceberg?

1.  **Интенсивные Upserts**: Hudi изначально создавался для сценариев с частыми обновлениями и вставками. Его механизмы индексации позволяют очень быстро находить файлы для обновления, минимизируя Write Amplification.
2.  **Streaming Ingestion**: Hudi отлично оптимизирован для потоковой загрузки данных с низкой задержкой.
3.  **Merge-On-Read**: Если нужна минимальная задержка между записью и чтением, режим MOR в Hudi позволяет быстро писать данные в лог-файлы и мержить их при чтении, не переписывая сразу Parquet-файлы.

### Когда оправдано использовать Delta Lake вместо Iceberg?

1.  **Экосистема Databricks**: Если вы уже используете платформу Databricks, Delta Lake является нативным и наиболее оптимизированным форматом .
2.  **Z-Ordering и Data Skipping**: Delta Lake имеет мощные встроенные механизмы оптимизации размещения данных , которые могут значительно ускорить запросы с фильтрацией по нескольким колонкам.
3.  **Простота использования**: Delta Lake часто проще в настройке и использовании "из коробки" для пользователей Spark, так как не требует сложной конфигурации каталогов.

### Общий вывод

*   **Iceberg**: Отличный выбор как открытый стандарт, независимый от вендора, с хорошей производительностью чтения и записи для больших аналитических таблиц.
*   **Delta Lake**: Лидер по производительности в экосистеме Spark/Databricks, мощные фичи оптимизации.
*   **Hudi**: Специализированный инструмент для CDC, частых апдейтов и потоковой обработки данных.
